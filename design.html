Summarize this website in a sentence?

Peak Impact from Minimum Complexity.

My goal with this website was to make something that is both simple but impactful at the same time.

My two main inspirations: Apple and Comp Con.

Design Process:

Visual Design Elements:

For my website, I'm using a Google font known as Quicksand, which gives the site a professional feel while still keeping things approachable thanks to a solid line weight but a lack of sharp edges. I increased the standard font size across the whole website, since I felt that the standard font is too small and hard to read, and is thus an accessibility issue that I needed to solve.

FONT sample

For my color palette, I have two: A UI palette, which consists of whitesmoke, darkslategray, gray and black, and a page palette, which I use to break the site up into distinctive blocks. This second palette consists of four colors: Aquamarine, Coral, Peachpuff and Lightskyblue. All of these colors are bright enough to give the site some vibrancy, but not so bright as to overwhelm the user. My primary inspiration for doing this was the Comp/Con companion app for the TTRPG Lancer, which is pictured above. Comp/Con is more dynamic with animations and brighter colors, but I liked the idea of using impactful colors to make the site visually appealing. I wanted to make something that is pleasant to look at, without being too minimal or maximal, and the different colors also block out the site really well and make it easy to see where different parts of it begin and end.

For the shapes used in my site, I generally stuck with rounded edges, since those make the site less aggressive and feel more personal with their friendlier feel. By doing this, the site can better serve as a portfolio that’s simply meant to be browsed through, rather than something formal that must be used purely to reach a goal before leaving as soon as you are done.

Implementation notes:

Making this website was very much a learning experience for me. When I started it, I knew nothing about HTML. Now I know a little more, but I’m still learning. The first and most important thing I did learn, however, is to bring over my most important lesson from every other bit of code I’ve ever written: make it scalable. If you look closely at this site and get into the actual HTML of it, you will quickly find that I have reused a lot of CSS and code throughout the site. Because it makes no sense to redo everything every time, especially when you’re trying to make a coherent website with a consistent style.

Besides the more modular approach I started taking to the website about 1/8th of the way through, the other thing of note I did throughout this phase of the project is I made an awesome piece of code that automatically builds galleries for my pages, using the code from one of our weekly exercises as a starting point. Doing this may have been a little work to set up, but it now means I can very easily and concisely build my galleries in other areas of my site as I need to while just using one script. It’s also a good bit of future proofing, since if I were to ever plug this site into a proper database, I could very easily pull galleries from that in future, which is something I did on the advice of my father who works in IT.

Moving into the second assignment, I've practically been a different designer. I've learnt a lot more about what HTML, JS and CSS can and can't do (not that there's anything they can't do, moreso they aren't great at everything) and more importantly I've become willing to learn from sources outside of Wits. MDN web docs have been hugely useful for this, but in general I've vastly expand my range of sources to deepen my learning and to allow me to build a faster, prettier and -most importantly- more responsive website that functions as intended on a range of devices.

And for the final assignment, I’ve noticed that my early design choices have really come back to haunt me, specifically with microformatting. Microformatting requires a very different design mindset, and I likely would have been able to implement far more of it if I had known to plan around it at the start. The same is true with semantic elements. Ultimately, web development is a space of constant learning, because while the basics are easy to grasp, you could potentially spend a lifetime tweaking a single page, nevermind building a whole site. I do think that I’m a lot better than when I started, but like the home page says, improvement is always, and I still have a long way to go.

IXD Process:

Goal Alignment:

Originally this was the simplest area for me, since my goal was to provide a website that can serve as an area to display my work and projects. As such, all I need the user to do is to be able to move around the site as they want to or need to.

However, as the assignment has gone on, my primary purpose for this site has become using it as a portable showcase for my work. That’s why I’ve put a lot of effort into mobile compatibility and why my work is always put first and foremost, because since bringing this site live I have started genuinely using it to show what I’ve done to friends and family, which is hugely useful to me as someone who is often too lackadaisical about showing the amount of work they actually put in on a daily basis.

A point of specific focus for me is the layout of content. The requirements for the site did give me a good starting point with that, but I still tried to lay my information out in a logical hierarchy to make the website easier to navigate. This also helps the user to know what information of mine is the most important, which should let them understand my intentions as the creator a little better and also keep anyone who’s casually using the site focused upon the areas I deem most important.

Information structure:

As mentioned before, I put a focus into this area. A big item of focus for me was my top bar. I wanted to follow a three click rule, which means that any area of my site should either be accessible from the top bar or from a page included in the top bar. This rule helped me to design a website that I think is easy to use, as well as responsive.

User Flow:

The website is designed to be easy to navigate, with the points of interaction always being highlighted using a consistent design language and a use of movement to make them stand out. This should make it easier for users to not just navigate my site, but also to utilize the features I’ve implemented, even if most of them are simple and don’t require too much interaction from the user.

Interface Elements:

All my interface elements follow a standardized style, which should make them easier to use and identify. Beyond the buttons to take the user to different parts of my site and the top bar that allows them to navigate the space as a whole, I have also added a hamburger menu to allow ease of navigation with reduced screen size, a footer with some basic functionality and a return to top button that allows the user to quickly return to the top of the page if they need to for any reason. In general, my interface is something I am particularly happy with, since it’s both functional and aesthetically unintrusive, which is the most any user interface can ask for when it is not part of the core experience itself.

Code impact on UI and UX:

My UI is heavily code driven now, thanks to my nifty javascript function that populates my top and side bars from an array instead of manually placing them in each page. This lets me very easily make site wide changes without needing to take hours to implement them, making it a very welcome addition. This is definitely the largest change, and I’d say the most impactful, since it means that the experience across my whole site is a lot more unified now, which should make it easier to learn and create a more pleasant experience overall.

Beyond the most important implementations, I’ve got a few others. The galleries and text sections of the site are both populated by Java, allowing me to keep the HTML clean and also to unify my various posts and entries with very little difficulty. Java also powers the useful function that allows me to display entire entries in their own windows, which is a major boon to the cleanliness of the site. The back to top button is also partially driven by Java, although less so than other aspects, and I have a search function that’s really more of a filter function that should hopefully make finding specific entries a little easier, although it’s not the greatest and is probably the part of the website that could use the most improvement.

Finally, I believe that UX is affected positively by the code overall. My site is a generally smooth experience to operate, even if there is room for improvement, and the code is a big part of that. Be that in terms of me polishing a lot of the surface level imperfections off through hard trial and error, or just the simple fact that a lot of my implementations make the site better, this site is a long shot from where it started, and I believe it’s a far more pleasant experience because of it. There’s still work to do, don’t get me wrong, but I don’t think any of my coding changes have made the site worse at least.
